1プロジェクト目的・背景（案）
①
⚡ 安定稼働の実現
現状ピーク時（スタジアム入場直後・大手LINE告知後）に落ちる抽選処理を改善し、同時1万アクセスでも安定稼働させる。おとなサントリーなどLINE友だち100万人超の告知にも耐えるシステムへ。
②
🛠️ 開発の手離れ実現
営業が新キャンペーンを自分で設定できるようにし、開発依頼をゼロに近づける。8画面以上に分散した管理UIをウィザード化し、専門知識不要で設定完走できる状態へ。
③
🔧 保守性・拡張性の確立
Perl 5make_perl（EOL）からPHP Laravelへ移行し、メンテナンス可能なメンバーを増やす。CI/CD・自動テスト・Secrets管理を整備し、継続改善できる基盤を作る。

🔴 現状の主要課題（AsIs）

APerl言語のためメンテナンスが不十分（Ubuntu 18.04 EOL、Perl 5.26make_perl固定）
B瞬間風速の集中アクセスに対応できない（LOCK TABLES + CGI 1req=1プロセスが根本原因）
C管理画面の操作手順が分かりづらく、CP設定に専門知識が必要
Dエントリー時に都度抽選処理→DB直列化のボトルネック
E在庫を出し切れずに確率を手動調整→景品表示法に抵触するリスク
F結果画面に戻れるパターンが分かりづらい（UX問題）
G不正対策が最低限（捨てアドドメイン登録＋reCAPTCHA のみ）
H保存期間180日の仕様課題
Iクライアント要望を満たせない差し込み画面の制約
JAI搭載機能がない（文章生成・抽選ロジック自動変更・当落画像生成）
K設置の手間がかかっている（営業・クライアント側）
LCDNが機能していない（Cache-Control: no-store で常時動的生成）

💡 競合から学ぶ改善ヒント
PKBから：クレーンゲームやガラガラなどユニーク演出の追加検討。マイレージ型（ポイント累積）はPKB・OWNLYが対応しておりSmartKujiに不足。今後の機能拡張候補として優先度を検討すること。
OWNLYから：SNS連携強化（X/Instagram対応）とUGC収集機能は差別化余地がある。クロワッサン・OWNLYのノーコードUI・デザインテンプレートの充実度が高く、SmartKujiのウィザード化と合わせてテンプレート提供も強化すること。

⚡ 抽選処理の高速化

テーブルロック（LOCK TABLES）を廃止。楽観ロック or Redis原子操作で同時抽選を非直列化

🏢 事業開発部の責任詳細（PHASE 0 確定事項）
① サービスの方向性
SmartKujiをどのようなサービスとして位置づけるか。対象クライアントの規模・業種・課題感。PHP版リニューアルで何を実現するのかのビジョン設定
② 差別化ポイントの定義
競合（クロワッサン・電通TONARIWA等）に対する優位性の特定と優先順位付け。クジラロジック・景品表示法対応・参加方式5種など「外せない強み」の確定
③ ターゲティング・ポジショニング
誰に・どんな課題解決として・どの価格帯で売るのかを定義。営業部が訴求すべき顧客像と訴求シナリオの基本方針を確定させる

15 大西が最優先で検討すべき設計課題 設計フェーズ最重要
以下の5領域は、バックエンド・インフラ・セキュリティの根幹を決める意思決定事項である。7月中旬の設計完了マイルストーンまでに各領域の方針を確定させ、スプレッドシート「技術設計・インフラ」シートに記録する。決定前に着手した実装は手戻りリスクが高い。
①
システム管理画面 vs 企業管理画面の構造すみ分け
全体のDB設計・権限設計・URL設計に直結する最上流の決定事項
🔑 決める必要がある主な問い
  • WWS（システム管理者）と顧客企業（一般管理者）で操作できる機能の境界はどこか
  • マルチテナント構造か、顧客ごとに独立したDBスキーマか
  • システム管理者のみが触れる設定（クジラロジック確率・シリアル大量発行・グローバル設定）と顧客が触れる設定の仕切り
  • 顧客の管理者の中での役割分離（スーパー管理者 / 一般担当 / 閲覧のみ）の粒度
  • 代理店経由の案件でのアクセス権設計（代理店→顧客の中間権限をどう扱うか）
⚠️ 決めないと起きるリスク
  • DBのテーブル設計・API設計が後から根本変更になる
  • URLルーティング設計の総やり直し
  • 権限チェックロジックの二重実装・漏れ
  • 顧客へのデータ漏洩リスク（他社CPが見える等）
📐 検討すべき設計案
例：/system/（WWS専用）・/admin/（顧客企業）・/api/v1/（共通API）の3層分離。LaravelのGate + Policyで権限を宣言的に管理。tenants テーブルでマルチテナント対応。
②
実装可能かつコアな機能要件の優先順位精査（フェーズ分け）
10/31リリースに入れる機能を確定させる唯一の機会
🔴 Phase 1（v1.0・10/31必達）
現行Perl版と同等の機能を全てPHPで動かすことを最低ライン。抽選エンジン再設計・5参加方式・5ゲーム演出・管理画面ウィザード化・非同期メール・AWS移行・セキュリティ基盤
🟠 Phase 2（v1.1・2027年Q1）
新規差別化機能。設定連動型プレビュー自動生成・シリアルコード自動発行・有料くじ（決済）・店舗分割CP・コース選択・商品詳細選択
🟢 Phase 3（v2.0・将来）
AI機能（文章生成・画像生成）・マイレージ型・SNS連携強化（X/Instagram）・UGC収集・レシートCP高度化・多言語対応
⚠️ 精査の視点：①Perl版で既に動いていること（移植）と②新規実装（リスク）を明確に分離する。新規機能は全てPhase 2以降に積極的に移す判断が10/31リリースの安全弁になる。
③
サーバー設計 ・ ドメイン・URL設計
インフラ費用・移行計画・SEO・既存顧客への影響に直結
🔑 サーバー設計の論点
  • ECS（常駐コンテナ）vs Lambda（サーバーレス）vs EC2の選択。コールドスタートと高負荷の両立
  • Aurora の Read Replica をどの処理に使うか（抽選結果参照・当選者一覧取得など）
  • Redis の用途分離（セッション / キャッシュ / 事前抽選結果キュー）
  • 現行Perl環境との並行稼働期間の設計（新旧切り替えタイミング・ロールバック計画）
  • SQS Dead Letter Queue とリトライ戦略（メール失敗時の処理）
🔑 ドメイン・URL設計の論点
  • 現行 kuji.smartpr.jp/{cp_id}/ との互換性維持か、新URL体系か（既存QRコード・印刷物への影響）
  • 管理画面・参加者画面・APIのサブドメイン分割設計（例：admin.kuji.jp / cp.kuji.jp / api.kuji.jp）
  • 顧客独自ドメイン対応（cp.client.co.jp）のためのALB + 証明書管理設計
  • RESTful API バージョニング（/api/v1/）の将来移行計画
④
セキュリティ設計
ISMS・Pマーク維持・景品表示法準拠・不正対策の全てに関わる
🔐 認証・認可設計
  • Laravel Sanctum（SPA向け）vs Passport（OAuth2）の選択
  • 管理者ログインの2要素認証（2FA）要否
  • セッション固定化攻撃・セッションハイジャック対策
  • API認証トークンの有効期限・リフレッシュ設計
🛡️ 不正参加・シリアル不正対策
  • シリアルコードの総当たり攻撃対策（レート制限・ロックアウト）
  • 同一IPからの大量エントリー検知・遮断（WAF連携）
  • 捨てアドレスドメインブロックリストの自動更新
  • reCAPTCHA v3 → v3 Enterprise への移行検討
  • ワンタイムURLの署名検証・有効期限
🔒 データ保護・法令対応
  • AWS Secrets Manager への全認証情報移行（db.pl廃止）
  • 当選者個人情報の暗号化保存・保存期間ポリシー
  • ISMS継続審査に向けたログ管理・監査証跡設計
  • 景品表示法準拠：在庫出し切りの自動担保設計（確率0化 vs 在庫フラグ）
  • XSS / CSRF / SQLインジェクション / SSRF 対策の網羅的チェックリスト
⑤
負荷対策
目標：同時10,000アクセス・p95レイテンシ2秒以内・1,000 RPS負荷試験クリア
🏗️ アーキテクチャレベルの負荷対策
  • LOCK TABLES廃止の代替設計：Redis SET NX EX による原子的在庫確保 or 楽観ロック（バージョン番号による競合検知）のどちらを採用するか決定する
  • 事前抽選結果生成のタイミング：CP公開時・在庫追加時・バッチ夜間生成のいずれにするか。生成失敗時のフォールバック設計
  • Aurora Read Replica の活用範囲：当選者一覧・分析系クエリをReplicaへ向けるルーティング設計
  • Auto Scaling のトリガー設計：CPU使用率 vs RPS vs SQSキュー深度の組み合わせ
🧪 負荷試験設計
  • 試験シナリオ：LINE告知後の瞬間スパイク（1分で10,000接続）・持続負荷（30分 1,000 RPS）・シリアル認証集中
  • ツール選定：k6 / Locust / JMeterのいずれか。CI連携による回帰試験の組み込み
  • 合格基準の定義：エラー率 <0.1%・p95レイテンシ <2秒・DB接続エラーゼロ
  • キャッシュ戦略：CP設定のRedisキャッシュ有効期限・CloudFront TTL設定・キャッシュバスティング設計
  • コネクションプール設計：PHP-FPM × Aurora の最大コネクション数上限の計算と設定値の決定
📌 試験タイミング：9/11 STGテスト開始と同時に負荷試験を実施。10/2 FB対応完了時点で合格基準を満たすこと。
